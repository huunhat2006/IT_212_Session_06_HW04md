PHẦN 1 - LEVEL-BASED EXPLANATION
Cấp độ 1 - Beginner (Góc nhìn cuộc sống)
Hãy tưởng tượng bạn bước vào hai nhà hàng khác nhau đại diện cho hai mô hình xử lý:

Nhà hàng Blocking (Kiểu truyền thống - RestTemplate): Bạn (Request) đến bàn và gọi món. Một nhân viên phục vụ (Thread) tiếp nhận đơn của bạn. Điểm mấu chốt là: Người phục vụ này phải đứng im tại bàn của bạn, chờ nhà bếp nấu xong (I/O Block), rồi bê món ra cho bạn. Trong suốt thời gian món đang nấu, người phục vụ đó không thể đi đón khách mới. Nếu nhà hàng có 100 người phục vụ, và cả 100 người đều đang đứng chờ bếp nấu cho 100 khách, thì vị khách thứ 101 buộc phải đứng ngoài cửa đợi, cho dù trong nhà hàng còn rất nhiều bàn trống.

Nhà hàng Non-blocking (Kiểu Reactive - WebClient):
Bạn đến bàn và gọi món. Người phục vụ tiếp nhận đơn, gửi ngay vào nhà bếp, đưa cho bạn một thiết bị báo rung (Paging Device / Mono-Flux) rồi lập tức quay sang đón vị khách tiếp theo. Nhà bếp nấu xong món nào sẽ bấm nút làm thiết bị của khách đó rung lên. Bạn tự ra quầy nhận món.

Tại sao Non-blocking xử lý được nhiều request hơn?
Vì người phục vụ không tốn thời gian "chết" để đứng chờ. Chỉ với một nhóm nhỏ nhân viên (ít Thread), nhà hàng Non-blocking vẫn có thể đón tiếp hàng ngàn khách cùng lúc mà không bị tắc nghẽn ở cửa ra vào.

Cấp độ 2 - Senior Developer (Góc nhìn chuyên sâu)
Trong Spring WebFlux, WebClient mặc định sử dụng Netty làm công cụ mạng (Network Engine) cốt lõi dưới nền tảng.

Reactive Programming & Reactive Streams: Là mô hình lập trình hướng đến các luồng dữ liệu bất đồng bộ (Asynchronous Data Streams). Nó tuân thủ chuẩn Reactive Streams gồm 4 interface chính: Publisher, Subscriber, Subscription, và Processor.

Mono & Flux (Publishers): * Mono<T>: Đại diện cho một luồng bất đồng bộ phát ra 0 hoặc 1 phần tử.

Flux<T>: Đại diện cho một luồng bất đồng bộ phát ra 0 đến N phần tử.

Event Loop Architecture (Netty): Thay vì tạo một Thread cho mỗi request (Thread-per-request), Netty chỉ tạo một số lượng nhỏ các Event Loop Thread (thường bằng số lượng CPU Cores ×2). Các Event Loop này chạy một vòng lặp vô hạn, liên tục lắng nghe và bóc tách các sự kiện (Event) từ các kênh mạng (Socket Channels).

Backpressure (Áp lực ngược): Cơ chế cho phép Subscriber (bên tiêu thụ dữ liệu) thông báo ngược lại cho Publisher (bên phát dữ liệu) về số lượng phần tử mà nó có khả năng xử lý tại một thời điểm, ngăn chặn tình trạng bên tiêu thụ bị quá tải ("ngập lụt" bộ nhớ).

PHẦN 2 - COMPARATIVE ANALYSIS
Bảng so sánh: RestTemplate vs WebClient
Tiêu chí	RestTemplate	WebClient
Mô hình xử lý	Blocking (Đồng bộ, nghẽn Thread)	Non-blocking (Bất đồng bộ, Reactive)
Số lượng Thread	Rất lớn (Tương đương số CCU)	Rất ít (Thường cố định bằng số CPU Cores)
Mức sử dụng RAM	Cao (Mỗi Thread tốn ~1MB cho Stack)	Thấp và ổn định (Ít tốn chi phí Thread Stack)
Throughput	Bị giới hạn bởi Thread Pool	Cực cao nhờ tối ưu hóa thời gian chờ I/O
Scalability	Khó scale (Chạm ngưỡng Thread Limit)	Dễ dàng scale (Mở rộng tuyến tính cực tốt)
Phù hợp Microservices	Kém (Dễ gây hiệu ứng đổ vỡ dây chuyền)	Rất tốt (Lý tưởng cho mô hình gọi chéo API)
Xử lý 10,000 CCU	Quá tải / Sập hệ thống (Thread Starvation)	Xử lý mượt mà (Tài nguyên tăng không đáng kể)
Phân tích kịch bản: 10,000 Request đồng thời (CCU)
Giả định mỗi hệ thống gọi sang một API bên thứ ba mất trung bình 1 giây để phản hồi.

Với RestTemplate (Mô hình Spring MVC truyền thống):

Số Thread cần thiết: Hệ thống cần gần 10,000 Threads hoạt động cùng lúc để phục vụ 10,000 khách hàng này.

Tiêu thụ tài nguyên: Phá sản về tài nguyên. 10,000 Threads ×1MB Stack Size≈10GB RAM chỉ dành riêng để duy trì các Thread sống. Thêm vào đó, CPU sẽ bị quá tải nghiêm trọng do hiện tượng Context Switching (CPU liên tục phải nhảy qua nhảy lại giữa 10,000 luồng để kiểm tra xem có dữ liệu về chưa). Hệ thống sẽ rơi vào trạng thái Thread Starvation và sập (Crash).

Với WebClient (Mô hình Spring WebFlux):

Số Thread cần thiết: Chỉ cần khoảng 8 đến 16 Threads (Tùy thuộc vào số nhân CPU của server).

Tiêu thụ tài nguyên: Cực kỳ nhỏ. 10,000 connection mạng được quản lý dưới dạng các file descriptor (Socket Channels) bởi kiến trúc Selectors của Epoll/Kqueue trong Netty. Mức tiêu thụ RAM tăng thêm rất ít (chỉ vài chục Megabytes để chứa metadata của connection). CPU hoạt động hiệu quả vì không phải đổi ngữ cảnh (Context Switching), tận dụng tối đa thời gian để xử lý dữ liệu thực tế.

package com.speedycart.integration.service;

import io.netty.channel.ChannelOption;
import io.netty.handler.timeout.ReadTimeoutHandler;
import io.netty.handler.timeout.WriteTimeoutHandler;
import lombok.Getter;
import lombok.Setter;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.HttpStatusCode;
import org.springframework.http.client.reactive.ReactorClientHttpConnector;
import org.springframework.stereotype.Service;
import org.springframework.web.reactive.function.client.WebClient;
import reactor.core.publisher.Mono;
import reactor.netty.http.client.HttpClient;
import reactor.netty.resources.ConnectionProvider;
import reactor.util.retry.Retry;

import java.io.IOException;
import java.net.ConnectException;
import java.time.Duration;
import java.util.concurrent.TimeUnit;

@Slf4j
@Service
public class DeliveryIntegrationService {

    private final WebClient webClient;

    // Cấu hình cứng theo yêu cầu kỹ thuật hệ thống
    private static final int TIMEOUT_DURATION_MS = 5000; 
    private static final long MAX_RETRY_ATTEMPTS = 3;
    private static final int BACKOFF_FIRST_RETRY_SECONDS = 1;

    public DeliveryIntegrationService(WebClient.Builder webClientBuilder) {
        // 1. Cấu hình Connection Pool cho Netty HttpClient để chạy Production
        ConnectionProvider connectionProvider = ConnectionProvider.builder("deliveryPool")
                .maxConnections(500) // Tối đa 500 connection song song
                .pendingAcquireTimeout(Duration.ofSeconds(20))
                .maxIdleTime(Duration.ofMinutes(5))
                .build();

        HttpClient httpClient = HttpClient.create(connectionProvider)
                .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, TIMEOUT_DURATION_MS) // Connection Timeout
                .doOnConnected(conn -> conn
                        .addHandlerLast(new ReadTimeoutHandler(TIMEOUT_DURATION_MS, TimeUnit.MILLISECONDS))   // Read Timeout
                        .addHandlerLast(new WriteTimeoutHandler(TIMEOUT_DURATION_MS, TimeUnit.MILLISECONDS))); // Write Timeout

        this.webClient = webClientBuilder
                .baseUrl("https://api.delivery-partner.com/v1") // Domain bên vận chuyển giả định
                .clientConnector(new ReactorClientHttpConnector(httpClient))
                .build();
    }

    /**
     * Gửi thông tin đơn hàng sang bên đơn vị vận chuyển đối tác.
     * @param orderRequest Đối tượng DTO chứa dữ liệu đơn hàng
     * @return Mono<DeliveryResponse> Kết quả trả về bất đồng bộ
     */
    public Mono<DeliveryResponse> sendOrderToDelivery(DeliveryOrderRequest orderRequest) {
        if (orderRequest == null) {
            return Mono.error(new IllegalArgumentException("Dữ liệu đơn hàng đầu vào bị null."));
        }

        String orderId = orderRequest.getOrderId();
        log.info("[DELIVERY_START] Bắt đầu gửi thông tin đơn hàng sang API vận chuyển. OrderId: {}", orderId);

        return this.webClient.post()
                .uri("/orders")
                .bodyValue(orderRequest)
                .retrieve()
                // Xử lý khi API bên thứ ba trả về mã lỗi 4xx hoặc 5xx
                .onStatus(HttpStatusCode::isError, clientResponse -> clientResponse.bodyToMono(String.class)
                        .flatMap(errorBody -> {
                            log.error("[DELIVERY_API_ERROR] Phía vận chuyển trả về lỗi. Code: {}, Body: {}", 
                                    clientResponse.statusCode(), errorBody);
                            return Mono.error(new RuntimeException("API vận chuyển báo lỗi hệ thống."));
                        }))
                .bodyToMono(DeliveryResponse.class)
                // Cấu hình cơ chế Retry phòng thủ
                .retryWhen(Retry.backoff(MAX_RETRY_ATTEMPTS, Duration.ofSeconds(BACKOFF_FIRST_RETRY_SECONDS))
                        .filter(this::isTransientException) // Chỉ retry các lỗi đường truyền mạng hoặc timeout
                        .doBeforeRetry(retrySignal -> log.warn("[DELIVERY_RETRY] Tiến hành gửi lại đơn hàng lần thứ {}. OrderId: {}. Lý do lỗi: {}", 
                                retrySignal.totalRetries() + 1, orderId, retrySignal.failure().getMessage()))
                        .onRetryExhaustedThrow((retryBackoffSpec, retrySignal) -> {
                            log.error("[DELIVERY_FATAL] Đã cạn kiệt số lần thử lại ({}), gửi đơn hàng thất bại cuối cùng. OrderId: {}", 
                                    MAX_RETRY_ATTEMPTS, orderId);
                            return new RuntimeException("Thất bại hoàn toàn khi đồng bộ sang đơn vị vận chuyển.", retrySignal.failure());
                        }))
                // Log khi luồng phát tín hiệu thành công (onNext)
                .doOnNext(response -> log.info("[DELIVERY_SUCCESS] Gửi thông tin đơn hàng sang đối tác thành công. OrderId: {}, TrackingId: {}", 
                        orderId, response.getTrackingNumber()));
    }

    /**
     * Bộ lọc xác định xem ngoại lệ có thuộc diện được phép Retry theo yêu cầu bài toán không.
     */
    private boolean isTransientException(Throwable throwable) {
        // Kiểm tra xem lỗi có phải do Timeout (Read/Connect Timeout của Netty), IO, hoặc lỗi kết nối mạng không
        boolean isTimeout = throwable instanceof io.netty.handler.timeout.TimeoutException;
        boolean isConnectException = throwable instanceof ConnectException;
        boolean isIOException = throwable instanceof IOException;
        
        if (isTimeout) {
            log.error("[DELIVERY_TIMEOUT] Phát hiện sự cố Timeout mạng.");
        }
        
        return isTimeout || isConnectException || isIOException;
    }
}

PHẦN 4 - BEST PRACTICES & CONCLUSION
Kỹ sư WebFlux cần ghi nhớ (Best Practices):
Blockhound Integration: Luôn tích hợp thư viện BlockHound trong môi trường kiểm thử (Test Environment) để phát hiện và ngăn chặn lập tức bất kỳ đoạn code mã hóa đồng bộ (Thread.sleep(), JDBC, File IO truyền thống) nào vô tình lọt vào chạy trên Event Loop Thread. Một dòng code blocking có thể làm đóng băng toàn bộ hệ thống WebFlux của bạn.

Chỉ Subscribe ở tầng cuối cùng: Tuyệt đối tránh tự ý gọi .block() hoặc .subscribe() ở tầng Service. Hãy để Spring WebFlux tự quản lý luồng dữ liệu và subscribe ở tầng giao tiếp mạng cao nhất (Reactive Controller).

Cấu hình Connection Pool rõ ràng: Mặc định Netty tự tạo pool co giãn, tuy nhiên trên Production cần cấu hình tường minh thuộc tính maxConnections và pendingAcquireTimeout để kiểm soát chặt chẽ lượng tài nguyên kết nối ra ngoài, tránh cạn kiệt tài nguyên hệ thống.
